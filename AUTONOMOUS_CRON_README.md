## Architecture

```
Google Cloud Scheduler (timer)
          ↓
          HTTP POST
          ↓
   ┌─────────────────────────┐
   │  Cloud Function OR       │
   │  Cloud Run Cron Endpoint │
   │  (/cron endpoint)        │
   └─────────────────────────┘
          ↓
   ┌─────────────────────────┐
   │  cron_service.py        │
   │  batch_process_all_users│
   └─────────────────────────┘
          ↓ (per user)
   ┌─────────────────────────┐
   │ run_proactive_check()   │
   │ - Fetch tasks           │
   │ - Fetch calendar        │
   │ - Horizon agent         │
   │ - Alert generation      │
   │ - Firebase logging      │
   └─────────────────────────┘
```

## Production Deployment

### Recommended: Google Cloud Functions

```bash
# 1. Create function directory
mkdir aria-cron && cd aria-cron
cp ../cron_service.py main.py
cp ../agents.py .
cat > requirements.txt << 'EOF'
firebase-admin>=6.5.0
google-api-python-client>=2.129.0
google-auth-oauthlib>=1.2.0
google-auth-httplib2>=0.2.0
pytz>=2024.1
python-dotenv>=1.0.0
EOF

# 2. Deploy function
gcloud functions deploy aria-cron-endpoint \
  --runtime python310 \
  --trigger-http \
  --allow-unauthenticated \
  --entry-point cron_handler \
  --memory 512MB \
  --timeout 540s \
  --region us-central1

# 3. Create Cloud Scheduler job
gcloud scheduler jobs create http aria-cron-hourly \
  --location us-central1 \
  --schedule "0 * * * *" \
  --uri "https://us-central1-YOUR_PROJECT.cloudfunctions.net/aria-cron-endpoint" \
  --http-method POST

# 4. Test
gcloud scheduler jobs run aria-cron-hourly --location us-central1
```

See [CLOUD_SCHEDULER_DEPLOYMENT.md](CLOUD_SCHEDULER_DEPLOYMENT.md) for details.

## What Happens When Cron Fires

### Execution Flow

1. **Cloud Scheduler** sends HTTP POST to cron endpoint
2. **cron_handler()** receives request
3. **batch_process_all_users()** iterates through all users in Firebase
4. For each user:
   - Fetch their tasks from Firestore
   - Fetch their Google Calendar events
   - Run horizon agent to assess schedule density
   - Run `check_proactive_interventions()` to generate alerts
   - Log results back to Firebase
5. Return summary JSON with all alerts generated

## Integration with Streamlit App

The Streamlit app (`app.py`) **no longer needs** the `?cron=true` query parameter trick. 

The old code:
```python
if st.query_params.get("cron") == "true":
    # run check...
```

**Has been replaced** with a comment directing to the new service.

The Streamlit app still runs normally and can show alerts that were generated by the autonomous cron service.

## Monitoring & Debugging

### View Cron Logs

**Option 1: Cloud Functions**
```bash
gcloud functions logs read aria-cron-endpoint --limit 50
```

**Option 2: Cloud Scheduler**
```bash
gcloud scheduler jobs describe aria-cron-hourly --location us-central1
```

**Option 3: Check Firestore**
```
Firebase Console → Collections → user_meta
→ Each user document has "cron_last_run" timestamp
```

### Common Issues

| Issue | Solution |
|-------|----------|
| "Function timed out" | Increase timeout to 540s (9 min) |
| "Credentials not found" | Ensure Cloud Function has correct IAM roles |
| "No alerts generated" | Check if users have tasks in Firestore |
| "Calendar fetch fails" | Verify Google Calendar API is enabled |

## Technical Details

### Key Functions in cron_service.py

| Function | Purpose |
|----------|---------|
| `cron_handler(request)` | HTTP entry point for Cloud Functions |
| `batch_process_all_users()` | Iterate all users and run checks |
| `run_proactive_check_for_user(user_id)` | Run full check for one user |
| `fetch_user_tasks(user_id)` | Get tasks from Firestore |
| `fetch_user_calendar_events(user_id)` | Get calendar via Google API |

### Environment Variables Required

- `GEMINI_API_KEY` — For agent reasoning
- `GOOGLE_CALENDAR_CREDENTIALS` — (auto-managed via token.json in Firestore)
- `FIREBASE_CREDENTIALS` — (Cloud Functions auto-detects)
